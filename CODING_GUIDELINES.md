# Coding Guidelines (Go backend)

## Общие принципы

- Следуем принципам SOLID и чистой архитектуры.
- Бизнес‑логика не живёт в HTTP‑handlers и репозиториях, только в сервисах/домене.
- Предпочитаем простые, понятные решения сложным абстракциям.

## Go стиль

- Обязателен `go fmt` и `go vet`.
- Используем `golangci-lint` с базовым набором проверок.
- Именование:
  - интерфейсы: `<Name>Repository`, `<Name>Service` и т.п.
  - контекст: первый аргумент `ctx context.Context`.
- Ошибки:
  - возвращаем `error`, не паникуем (кроме truly fatal).
  - оборачиваем через `fmt.Errorf("context: %w", err)` или пакет ошибок, если используется.

## Структура пакетов

- Никаких циклических зависимостей.
- `pkg` — переиспользуемые утилиты без бизнес-логики (response, env и т.п.). `internal` не импортирует `pkg` для доменной логики, только для HTTP/инфраструктуры.
- Feature-based: каждая фича — пакет `internal/<feature>` (auth, users и т.п.).
- В feature-модуле: handler, service, repository, dto, errors — чёткое разделение по файлам.
- Общие сущности — в `internal/domain`.

## HTTP‑слой

- Handlers (в `internal/<feature>/handler.go`):
  - минимум логики — парсинг запросов, вызов сервиса, маппинг ошибок в HTTP‑ответы.
  - валидация через `binding` (gin) на DTO из `dto.go`.
  - **Swagger**: при добавлении/изменении эндпоинта обязательно обновить swagger-аннотации (`@Summary`, `@Description`, `@Param`, `@Success`, `@Failure`, `@Router`, `@Security`) и выполнить `make swagger`.
- Middleware (в `internal/http/middleware`):
  - авторизация (JWT), логирование, recovery.

## Repository‑слой

- Интерфейс в `repository.go`, реализация в `repository_postgres.go`.
- Не протаскиваем детали БД наружу (никаких `sql.Rows` или ORM‑специфики выше).
- Методы репозиториев работают с доменными моделями (`internal/domain`).

## Тесты

- Для нового сервиса или нетривиальной функции — обязательные unit‑тесты.
- Моки репозиториев — в `*_test.go` того же пакета.
- Никаких “комментариев‑тудушек” вместо тестов.

## Swagger

- API документируется через swag-аннотации в handlers.
- При любом изменении API (новый эндпоинт, изменение контракта) — обновить аннотации и выполнить `make swagger`.
- Swagger UI доступен по адресу `/swagger/index.html` при запущенном сервере.

## Требования к коду от AI‑агента

- Агент обязан:
  - соблюдать эту архитектуру и naming.
  - при изменении API — обновлять swagger-аннотации и перегенерировать docs (`make swagger`).
  - описывать план изменений перед тем, как предложить код.
  - показывать только релевантные изменения, без несвязанных рефакторингов.
- Если для реализации фичи нужно изменить правила, сначала предложить обновление этого файла.
