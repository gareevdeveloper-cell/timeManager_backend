# Работа с AI‑агентом для backend

Этот файл описывает, как формулировать задачи для AI‑агента (Cursor / VS Code и т.п.) в контексте backend‑проекта.

## Общие правила

- Всегда указывай, что:
  - архитектура описана в `ARCHITECTURE.md`
  - домен — в `DOMAIN.md`
  - правила кода — в `CODING_GUIDELINES.md`
  - API для фронтенда — в `docs/API_SPEC.md`
- Не проси “переписать всё”; ставь конкретные задачи.
- Проси сначала план, потом код/патчи.

## Шаблон промпта для новой фичи

> Контекст:  
> Мы работаем в backend‑репозитории (Go). Архитектура описана в `ARCHITECTURE.md`, домен — в `DOMAIN.md`, правила кодирования — в `CODING_GUIDELINES.md`.  
>  
> Задача:  
> Реализовать <описание фичи> для сущностей <...>.  
>  
> Требования:  
> - Соблюдать слои (domain/service/repository/http).  
> - Не менять существующие контрактные эндпоинты без необходимости.  
> - Покрыть бизнес‑логику тестами.  
> - При изменении API — обновить swagger-аннотации и выполнить `make swagger`.  
>  
> Формат ответа:  
> 1. Краткий план шагов.  
> 2. Список файлов, которые нужно создать/изменить.  
> 3. Патчи (diff) по этим файлам.  
> 4. Команды для запуска тестов/линтеров.

## Шаблон промпта для рефакторинга

> Контекст:  
> Сервис `internal/<feature>/service.go` стал слишком сложным. См. архитектуру и домен в `ARCHITECTURE.md` и `DOMAIN.md`.  
>  
> Задача:  
> 1. Объясни текущую структуру этого сервиса и его зависимости.  
> 2. Предложи план рефакторинга, не нарушая архитектурные правила.  
> 3. Покажи патчи для аккуратного рефакторинга (по шагам).  

## Проверка результата

После любых изменений агента:

```bash
go test ./...
golangci-lint run ./...
```

При изменении API (новые/изменённые эндпоинты):

```bash
make swagger
```
